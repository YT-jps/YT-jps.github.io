写c关于文件与命令的流程

发表于2026-06-18|更新于2026-06-18
|浏览量:
今天是本学期最后一次上课了，又上了个早八，回来舒服地吃个饭睡个回笼觉就一觉睡到将近下午三点了。简单写了下算法课的作业，又玩了玩，想了想，决定还是把前面写c语言的整个流程梳理一下，为以后出现意外的时候回过头来看。
不管怎么说，如果是电脑重启或关机后，自然需要先打开WSL，即先win+r打开命令框，输入powershell进入命令小黑框，输入wsl -d Ubuntu，即打开WSL。当然，即便没有关机，也可以偶尔输入wsl -l -v来判断一下WSL是否已经打开了，查看state字样，如果是running就说明WSL正在运行，如果是Stopped字样就说明没有打开，需要输入wsl -d Ubuntu来打开。如果有必要，可以输入wsl --shutdown来关闭WSL，但通常都没有必要关，除非遇到电脑卡顿或WSL出现比较诡异的情况。
在打开了WSL后，就可以着手准备写代码了：第一步自然就是打开VSCode，并且左下叫点类似于<>的按键选择Connect to WSL，即远程连接已经打开了的WSL。如果是涉及之前写过且保存了的代码，就直接打开相应文件（如果在VSCode图形化工具页面找不到也可以直接搜路径：/home/blackgoat/c-projects/项目名，反正所有c文件必然在c-projects文件夹里面）然后编辑代码即可。如果是想新建一个代码文件，则需要在终端输入mkdir -p ~/c-projects/项目名即可，这只是创建了后面准备包含c文件的文件夹，真正的c文件可以通过图形化工具来直接创建，但为了统一方便，我还是准备以后都采用终端命令行里输入：touch 文件名.c，当然，通常这里c文件名就直接命名为main，也就是说通常直接输入touch main.c即可。

这里简单说明一下powershell、WSL和VSCode的关系：1.powershell：整个windows系统的终端，在这里相当于就是WSL的管理器，负责开启关机或更新WSL等。2.WSL：其本质是微软在 Windows 内核里内置了一个轻量级虚拟机，能无缝地运行一整套真正的 Linux 操作系统，由三大块构成：一、虚拟化层：微软专门编译、自带的一个完整 Linux 内核，用来执行的所有 Linux 命令，比如 ls、gcc，发出的每一个指令，都是由这个真实的 Linux 内核处理的。所以才能跑 Valgrind、GDB 这些需要深入内核协作的底层工具。二、存储器，也就是放在D盘总的磁盘（硬盘），装着你Ubuntu 的整个文件系统（根目录、家目录、/usr/bin 等），简单来说就是存储Linux内核的所有文件。三、互通层：在文件上能和我的windows系统文件互通，网络上和windows共享一个ip，在 Linux 里启动一个服务器程序，可以直接在 Windows 浏览器用 localhost 访问。VS Code 就是在 WSL 里启动了一个“远程服务器”程序，然后本地的 VS Code 用户界面去连接它。所以写代码的感觉是本地的，但编译运行却是远程 Linux 的。

在经历了创建文件后（或找到了之前创建的文件，现在需要重新编辑或更改的时候），自然而言就进行代码的撰写或改编，这里只需要在VSCode里写就好了。写完后记得保存，然后输入gcc -g -O0 -o 输出程序名 源文件.c进行编译，这里输出程序名是指编译后的二进制代码文件的名字，一般直接命名为main就好了，源文件就是写的c代买文件，按照前面的命名风格也就是main.c。那么这么看，其实每次编译的命令也几乎就是高度统一的，即是：gcc -g -O0 -o main main.c。在编译无报错后（如果遇到报错，具体情况我后面再总结如何看错误以及相应的改正，现在是整个流程输入就先不管这个）就可以正常运行了，也就是在终端里输入./程序名即可，当然和编译一样，按照我的代码风格，这里也就是./main。这里需要主要使用命令创建c文件、编译和运行都需要保证终端处于对应文件的位置，输入pwd可以查看当前处于哪个文件位置，也可以输入ls查看当前文件夹下有哪些文件，通常需要有对应的main文件。如果发现不处于对应文件位置，就可以输入cd ~/c-projects/文件名，就可以跳转到对应文件位置。一般也可以观察提示符号是否以~/c-projects/文件名结尾，不是的话就有很大可能不处于对应文件位置。
在学习数据结构的过程中，往往内存泄露会很危险也是需要着重关注的地方，在这里，我考虑使用Valgrind来检查是否存在内存泄露，尤其是在使用了动态分配内存的时候。在运行过了程序后，输入valgrind --leak-check=full ./main。随后终端就会显示各种检查结果，如果真的发生了内存泄露，会根据泄露数量对泄露进行编号，并分别形成等量的命令输出行，例如：
7 bytes in 1 blocks are definitely lost in loss record 2 of 4       这行就是说在编号为2的泄露共7个字节完全泄露（definitely）
 at 0x4850858: malloc (vg_replace_malloc.c:447)		         这行就是说明泄露对象被创建的来源，这里就是malloc动态分配内存所创建的
	by 0x4001201: createStudent (main.c:13)				         这行就是泄露对象实际被创建的函数及具体语句
	by 0x40012B9: main (main.c:28)					         这行就是泄露对象是从哪里开始被创建的

当然，像上述这样分别介绍了每一处的泄露后会有一个以LEAK SUMMARY为标题的总结，分别列举本次：绝对泄露（definitely）、间接泄露（indirectly）、可能泄露（possibly）、仍可访问（still）、被屏蔽的（suppressed）了多少字节。其中后面两个尤其是最后一个大多可以忽略，只要前面三个没有泄露就基本满足要求。后面还会有一段文字，但是就可以暂时忽略了。如果没有发生泄漏，就会输出：All heap blocks were freed – no leaks are possible。这也是每次写代码的内存管理目标。
在内存检查后，如果有必要（例如出现异常值或想观察某变量在程序中的动态变化）就可以使用断点命令。首先在终端输入命令：gdb ./main，显示gdb提示符即可，然后通常会现在主函数main这里设置一个断点，即break main，然后在需要观察的值被创建处设置一个断点，即break 行号，这里行号即是该变量被创建的代码行位置。如果设置错了断点，输入delete 断点号即可删除该断点，其中断点号会在每次设置断点的时候就顺便打印。不需要再设置其它断点的话就可以输入run让程序跑起来了，由于前面在主函数main这里设置了一个断点，程序就会先跑到main这里停止（会显示main函数第一行的代码，表示当前程序跑在这行代码前停住了），此时一般只需要输入continue即可让程序跑到下一个断点处，通常就跑到了我们需要观察的变量创建处，然后就可以输入next来让程序跑一行，如果有必要就可以输入print 变量名，终端就会显示该变量当前的值。以此类推，用next就可以继续一行一行跑代码，直到观察到变量值开始异常的地方，就可以停止去修改代码了。如果不想一直输入next来跑，可以输入watch 变量名，就可以让程序跑到该变量发生变化的时候停止并打印该变量新旧两个值。最后需要退出的时候输入quit即可退出gdb。
在以后，会涉及到比较底层的汇编层面，现在就简单提一下，输入objdump -d ./main | grep -A30 “

:”即可开始查看汇编层面的一些指令或代码。

总结完后发现，总结学到的内容其实也还蛮累的，不过相当值得，如果发现有错误记得提醒作者哟！
